The core insight

When AI output goes wrong, the root cause is almost never a dumb model. It's a gap in framing, context, or understanding that the AI couldn't supply on its own. These five habits close those gaps — systematically.

The rehash loop — the key failure mode

When AI keeps generating slight variations of the same wrong answer no matter how you adjust your prompt, the AI has exhausted the context you've given it. Stop prompting. Apply the habits. The loop is a signal, not a bug.

01
Habit one

Context

Paying attention to what information you supply to the model, trying to figure out what else it needs to know, and supplying it clearly — code, comments, structure, intent, constraints, and what you've already tried.

Thin context — fails
"Fix the bug in my code." No language, no framework, no expected behaviour, no constraints. AI fills gaps with assumptions — usually wrong ones.
Rich context — works
"This Express JWT middleware returns 200 on expired tokens instead of 401. The token is verified with jsonwebtoken v9. I've already tried catching the TokenExpiredError explicitly but it's not reached."

What "rich context" includes

  • ✓The relevant code snippet (not the whole file)
  • ✓Your intent — what it's supposed to do, not just what it does
  • ✓Constraints — performance, readability, team conventions, architecture
  • ✓What you've already tried — so AI doesn't repeat failed paths
  • ✓The domain — "this is a high-frequency trading system" changes everything
Watch for context drift

In a multi-turn conversation, understanding degrades. When output starts to feel off, ask the AI to restate what it thinks the code does. If the summary is wrong, you've found the drift — and the gap to fill.

02
Habit two

Research

Actively using AI and external sources to deepen your own understanding of the problem — running examples, consulting documentation, checking references to verify what's really going on. The goal is you getting smarter, not just collecting info to paste.

The key mindset shift

Most developers try to prompt their way out of a stuck situation. Research says: stop prompting, start understanding. The rehash loop isn't a prompt problem — it's a knowledge gap. Fill the gap first.

Research sources — AI is one, not the only one

Ask AI to explain itself

Ask it to explain what it just generated — not fix it. If the explanation is wrong, you've found the gap. This is the fastest first step.

Official documentation

AI training data may be stale on version-specific behaviour. Go to source for library APIs, framework versions, and breaking changes.

Your own codebase

How was this solved before? Existing patterns are context the AI will never have. The codebase is your most relevant research source.

Stack Overflow / GitHub Issues

For environmental problems — known bugs, version incompatibilities, edge cases the community has hit. Don't reinvent what's already documented.

Research feeds framing

What you learn through research becomes the raw material for a sharper problem definition. You can't frame a problem you don't understand. Research isn't a detour — it's the on-ramp to Habit 3.

03
Habit three

Problem Framing

Using the information you've gathered to define the problem more clearly so the model can respond more usefully. Most AI failures are framing failures — the AI solved the problem you described, not the problem you had.

Prompt engineering = requirements engineering

The challenges in writing good prompts are the same ones software teams have grappled with for decades: scoping, specifying functional and nonfunctional requirements, and communicating intent precisely. The skills you already use for architecture apply directly here.

The four elements of a well-framed prompt

What you want

A specific functional outcome — not a vague improvement. "Accept input as a string parameter" not "make it more testable."

Constraints

Performance, readability, team coding standards, architectural decisions. Nonfunctional requirements matter as much as functional ones.

What you don't want

Explicitly close off overengineered paths. "Don't add abstraction layers", "keep the public interface unchanged", "no new dependencies."

Definition of done

How will you know the answer is right? A concrete acceptance criterion prevents the AI from stopping too early or going too far.

Vague frame → overengineering
"Make this code more testable." AI adds interface, implementation, mock objects, dependency injection throughout. Valid pattern — wrong problem. A miniature framework where a simple refactor was needed.
Precise frame → targeted fix
"Refactor this method to accept a string parameter instead of reading from stdin, so I can pass test input directly. Don't change the class structure." One surgical change. No abstraction layers added.

04
Habit four

Refining

Iterating your prompts deliberately. This isn't about random tweaks; it's about making targeted changes based on what the model got right and what it missed, and using those results to guide the next step.

Vibe coding (random tweaks)
Prompt → output → "make it more X" → same answer rephrased → repeat. No diagnosis. Leads directly to the rehash loop. Code that runs but accumulates structural debt with each iteration.
Deliberate refining
Read output → diagnose what was right, what was specifically wrong → make one targeted change → re-prompt. Steers toward better patterns. Moves from "this works" to "this is actually good."

The deliberate refining loop — step by step

1
Read the output Don't skim. Understand what the AI actually produced before evaluating it.
2
Diagnose specifically: what was right? What was wrong? The error handling is right but business logic is in the controller — not "this is bad."
3
Make one targeted change to the prompt Address the specific gap you identified. One thing at a time.
4
Also refine the code — not just the prompt Clean names, reduce duplication, simplify logic, verify architecture fit. Don't stop when it compiles.
Agile parallel

Just as Agile shifted requirements from static specs to living conversations, prompt engineering shifts AI interaction from single-shot commands to iterative refinement — except you must infer what's missing from the output rather than having the AI ask you clarifying questions.

05
Habit five

Critical Thinking

Judging the quality of AI output rather than simply accepting it. Does the suggestion make sense? Is it correct, relevant, plausible? This habit is especially important because AI sounds confident even when it's wrong — and as models improve, the output becomes harder to question.

The core danger

AI doesn't truly understand your codebase. It mimics patterns from its training, producing answers that look right. It's very often correct — which is why vibe coding works at all. But it rarely accounts for overall architecture, long-term strategy, or good design principles. Those gaps are your responsibility.

Three concrete techniques

A
Generate alternatives Ask for 2–3 solutions. Comparing them reveals different assumptions and trade-offs — and keeps your own judgment engaged rather than passive.
B
Use AI as its own critic After it generates code, ask it to review that same code for problems or improvements. The context shift surfaces edge cases and design issues it didn't catch the first time.
C
Read, run, debug — then ask "will this make sense in 6 months?" Quick design review: Is coupling creeping in? Are names clear? Does this fit the architecture? Catching this now is far cheaper than finding it after it's woven through the codebase.

Signs the habit is taking hold in your team

  • ✓Asking "why did AI choose this pattern?" not just "does it work?"
  • ✓Relating AI output to readability, separation of concerns, testability
  • ✓Treating AI failures as learning opportunities — not blame
  • ✓Reviewing AI-generated code with the same rigour as human-written code
The long-term reality

More convincing output will require more sophisticated evaluation. Models will keep improving. What won't change is the need for developers to think critically about the code in front of them. As AI gets better, this habit becomes more important — not less.
